=== S1.2 AUTHORITATIVE W-FREEZE — real-machine acceptance (2026-07-08) ===
Plan: docs/refactor/15_signal_and_window_plan.md §1.2 ; ADR-0012 (+correction) in DECISIONS.md.

SETUP
- Cluster: isolated tre-v2 data plane (ADR-0008), the authoritative plane the freeze must run on.
- Controller image: tre-v2-controller:20260707-07717371.
- Controller mode: OBSERVE for the entire run (decisions computed + logged, NO fleet actuation).
  => Z_m read straight from the decision store while the fleet stays put (3 awake, 17 sleeping).
- Signal env under test: TRE_METRICS_WINDOW_MODE=sliding, TRE_METRICS_WINDOW_MS=30000,
  TRE_METRICS_REFRESH_INTERVAL_SECONDS=5, TRE_MIN_LATENCY_SAMPLES=10.
- Fleet: dsqwen-7b / dsllama-8b / dsqwen-14b each awake:1 (routable_pods=1), 17 sleeping. Unchanged before/after.

METHOD
- Step load driven from node10 host (same host clock as the redis/controller pods -> onset and
  decision ts share one clock; measured skew = one refresh cadence only).
- Per trial: (a) idle-guard — confirm the model's latest decision entry is idle (z_m null, trs 0);
  (b) record onset_ms = wall clock at first request; (c) fire a 20-worker saturating step load
  (max_tokens=128, input=64) for 60s; (d) 45s cooldown to drain the sliding window back to idle.
- z_m source: per-model decision-history zset tre:v2:decision:hist:<model> (scored by window_end_ms),
  dumped by score range after the run. Each entry carries z_m/trs/trs_z_m/state/signal_warm/window_end_ms.
- 12 trials = 4 rounds x {dsqwen-7b, dsllama-8b, dsqwen-14b}. Models pooled for statistics
  (plan15 §1.2 does not prescribe a per-model split; pooling noted here).

PRIMARY METRIC (apples-to-apples with the pre-change 60-120s number, which was the first-reflection
lag under the old 60s TUMBLING window):
  lag_first = t(first decision entry after onset with z_m != null) - t(onset)
  i.e. how long until the shared TSS signal first REFLECTS the added load.
  Doc bound: <= W + refresh + write granularity ~= 35s worst case at W=30000.

INTERPRETATION
- lag_first P95 = 12.7s (max 14.9s) is comfortably <= the 35s target and ~5-10x better than the
  pre-change 60-120s tumbling-window first-reflection lag. This is the acceptance quantity: the
  first moment Z_m reflects the load rise, bounded in theory by W + refresh + write (~35s worst case).
- lag_settle (44-52s) is the time for z_m to reach ~90% of its saturated plateau. It is inherently
  >= W and keeps climbing as the single replica's queue builds under sustained saturation; it is NOT
  the "reflect" latency the criterion gates on, and is reported only as context.

JUDGMENT
  lag_first P95 = 12.7s  <=  35s  =>  PASS.
  ==> W is AUTHORITATIVELY FROZEN at TRE_METRICS_WINDOW_MS = 30000 (sliding, 5s refresh).
  This is the single window S1.4 / R3 theta_m refit must use. The S1.4/R3 hard gate is now satisfied
  on the W side (S4 raw-logging remains its own precondition).
